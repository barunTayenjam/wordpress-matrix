## Identified Issues (pre-analyzed, verified against codebase)

### Issue 1: Unbound variables under `set -u` (CRASH)
`set -euo pipefail` is set at line 4. Functions that assign positional params without defaults crash:

- **Line 776:** `local force_yes="$2"` in `remove_site()` — crashes on `./matrix remove testsite` (no 2nd arg)
- **Line 418:** `local backup_file="$2"` in `restore_site()` — same pattern
- **Line 1442:** `local dump_file="$2"` in `import_db()` — same pattern
- **Line 1479:** `local output_file="$2"` in `export_db()` — same pattern

**Fix:** Use `${2:-}` default syntax.

### Issue 2: Container runtime detection uses wrong check (line 256-265)
`create_site()` checks `command -v docker` then `command -v podman` to decide which runtime to use. But when using podman-compose, `docker` command may not exist.

**Fix:** Derive `$CONTAINER_RUNTIME` from `$DOCKER_COMPOSE` once at detection time (after line 79). Use it everywhere.

### Issue 3: Hardcoded `docker` commands (13 occurrences)
Lines 36, 257, 464, 479, 525, 527, 697, 1140 all call `docker` directly.

**Fix:** Replace with `$CONTAINER_RUNTIME`.

### Issue 4: `$DOCKER_COMPOSE ps --format` incompatibility with podman-compose
podman-compose 1.5.0 may not support `--format "{{.Names}}"`. Used in ~15 places.

**Fix:** Create `compose_ps_names()` helper with fallback parsing.

### Issue 5: `clean_env` uses Docker-specific subcommands (lines 1518-1519)
`$DOCKER_COMPOSE system prune` falls back to `docker system prune`. Podman uses `podman system prune`.

**Fix:** Use `$CONTAINER_RUNTIME system prune -f`.

### Issue 6: `--profile tools` for phpcs/phpstan (lines 1045, 1056)
podman-compose 1.5.0 has limited/buggy `--profile` support.

**Fix:** Create `run_tool_container()` with compose-profile-first, direct-run-fallback strategy.

    ```bash
        local names
        names=$($DOCKER_COMPOSE ps --format "{{.Names}}" 2>/dev/null) || true
        if [[ -z "$names" ]]; then
            # podman-compose may not support --format, parse text output
    ```

    ```bash
    # Run a tool container with compose --profile, falling back to direct run for podman
    run_tool_container() {
        local service="$1"
        shift
        # Try compose profile first (works with docker-compose)
        if $DOCKER_COMPOSE --profile tools run --rm "$service" "$@" 2>/dev/null; then
            return 0
        fi
        # Fallback: direct container run (podman compatibility)
        case "$service" in
            phpcs)
                $CONTAINER_RUNTIME run --rm -v "$PROJECT_ROOT":/workspace:ro -w /workspace --network wp-net \
                    wpengine/phpcs:latest "$@"
                ;;
            phpstan)
                $CONTAINER_RUNTIME run --rm -v "$PROJECT_ROOT":/project:ro -w /project --network wp-net \
                    ghcr.io/phpstan/phpstan:latest "$@"
                ;;
            wp-cli)
                $CONTAINER_RUNTIME run --rm -v "$PROJECT_ROOT/wp_":/var/www/html:rw -w /var/www/html --network wp-net \
                    wordpress:cli "$@"
                ;;
            *)
                log_error "Unknown tool service: $service"
                return 1
                ;;
        esac
    }
    ```
